Skill 加载机制
通用模式

问题：Agent 的 skill 是怎么从代码变成可被调用的？加载时机和机制有哪些主流做法？

三种加载档位（按规模递进）

规模方式实现业界代表
< 10 个静态注册字典 / 装饰器，模块导入即就绪，O(1) 查找LangChain @tool、CrewAI Agent.tools=[...]
10 ~ 50 个动态发现entry_points / 文件系统扫描 / 插件目录，启动时加载VS Code 插件、Django apps
> 50 个RAG 检索把 tool description 向量化，按用户消息召回 top-KLangChain ToolRetriever / VectorStoreRetrieverTool

加载时机

  • 饿汉式（Eager）：服务启动时全部加载到内存，首调用零延迟，但启动慢
  • 懒汉式（Lazy）：首次调用时才加载，启动快但首调用慢；适合重型 / 低频 skill
  • 预热式（Warm-up）：启动后异步后台预加载高频 skill，兼顾两者

设计要点

  • 协议化：skill 应是声明式 schema（name / description / args_schema / permission），与具体实现解耦。LangChain 用 args_schema: Pydantic，Anthropic 用 JSON Schema
  • 权限模型：CrewAI 的 allow / ask / deny 三态权限，写操作必须 ask（人工确认）
  • 参数校验前置：在调用 LLM 后、执行副作用前用 Pydantic 校验，避免 LLM 幻觉参数炸穿下游
  • 白名单执行：LLM 返回 action name 后由代码查找注册表，不是 LLM 直接执行，未注册的直接拒绝
  • 失败可降级：skill 调用失败要有 fallback（默认动作 / 提示用户 / 转人工）
◆业界共识 渐进式披露（progressive disclosure）：不要一开始就把所有 skill 详情塞给模型，先让模型看到摘要，需要时再加载详情。这是 Claude Code 的核心设计哲学之一。

Claude 官方的加载机制
Anthropic 2026

问题：Claude Code 官方的 skill / 工具加载机制是什么？为什么这么设计？

截至 2026 年，Claude Code 实际上是一个 "七件套" 架构。2026 年 4 月 VILA Lab 的学术论文系统分析了它的 TypeScript 源码，结论是：核心永远是那几个解耦的扩展点。

Claude Code 七大组件

组件本质加载方式典型用途
SkillsMarkdown 文件按需触发（关键词匹配 / 模型主动调用）领域知识、操作 SOP、代码规范
MCP标准化协议Server 进程，stdio / SSE 通信接外部系统（DB / API / 文件系统）
Commands斜杠命令静态注册，/command 触发常用操作快捷方式

---
name: pdf-reader
description: Reads PDF files and extracts text/tables/images
---
# PDF Reading Skill

When the user asks to read a PDF:
1. Use the extract_pdf MCP tool
2. For tables, prefer the table_extract mode
3. Return markdown-formatted output

## Edge cases
- Encrypted PDFs: prompt user for password
- Scanned PDFs: route to OCR subagent

为什么这么设计（Anthropic 的设计哲学）

  1. 渐进式披露：模型一开始只看到 skill 的 name + description（几十 token），需要时才加载正文（可能上千 token）。避免一次塞几万 token 把上下文撑爆
  2. 文件即配置：Markdown 是最普遍的格式，门槛极低，非工程师也能写。文件即文档、即版本控制、即代码评审对象
  3. 协议化（MCP）：把"工具实现"和"模型如何调用"解耦。MCP server 可以是 Python / Node / Go 任意语言，Claude 只关心协议（JSON-RPC over stdio / SSE）
  4. 事件驱动（Hooks）：把横切关注点（审计 / 安全 / 自动化）从主流程剥离，避免主 prompt 被污染
  5. 子 Agent 隔离上下文：Subagent 有独立 context window，主 Agent 不会被冗长中间结果污染。对齐 AutoGen GroupChat 的设计
  6. 项目记忆固化（CLAUDE.md）：把团队约定、技术栈、规范固化为文件，避免每次会话重复声明。对齐 Cursor .cursorrules、Aider .aider.conf.yml
◆核心洞察 Anthropic 的设计反直觉之处：skill 不应该是代码。代码解决"怎么做"，但 Agent 的大问题是"什么时候做、按什么 SOP 做"，这是知识层面的问题，用 Markdown 表达最自然。MCP 解决"怎么做"的部分，两者正交分解。
◆SDK 演进 2026 年 5 月，Anthropic 把 Claude Code SDK 改名为 Claude Agent SDK——背后是判断：这不再是"帮你写代码的工具"，而是能自主读代码、改代码、跑测试的通用 Agent 平台。

生产系统 80-100 skill 该如何设计
大规模

问题：当 skill 数量到 80-100 这个量级，静态注册 + 全量塞 prompt 会爆，怎么设计？

!问题本质 100 个 skill 的 description 平均 100 token，全量塞 prompt = 1 万 token，还没算 schema。模型注意力会被稀释，选错概率随 skill 数量线性上升（参考 Anthropic 内部测试）。问题不是"内存不够"，是"注意力不够"。

三层路由架构（业界共识）

用户输入
→
L1 意图分类
Router LLM
max_tokens=200
→
L2 工具检索
RAG top-K=5~10
→
L3 执行
Pydantic + 白名单

L1 — 意图分类（收窄空间）

用一次便宜的 LLM 调用（小模型 / 降参数）把用户消息分到 5-10 个领域（如 strategy_config / auth / model_config）。这一步只看意图，不生成参数。

L2 — 工具检索（RAG）

对每个意图域，把该域内所有 skill 的 (name, description) 向量化，按用户消息召回 top-K（一般 5-10 个）。只有这 K 个 skill 的完整 schema 才会被塞进 prompt。

# 业界做法：LangChain ToolRetriever / VectorStoreRetrieverTool
from langchain.tools.retriever import ToolRetriever

retriever = vector_store.as_retriever(search_kwargs={"k": 8})
# Agent 只看到 8 个相关工具的描述，而不是 100 个
agent = AgentExecutor(tools=[], retriever=retriever, ...)

L3 — 执行（不变）

白名单 + Pydantic 校验 + 权限模型，和静态注册完全一致。L1/L2 解决"选哪个"，L3 解决"安全执行"。

技术作用典型实现

多智能体调度 · 是否使用状态机
Orchestration

问题：多智能体如何调度？业界是否使用状态机？

三大框架对状态机的态度

框架是否状态机调度模型特点
LangGraph显式状态图StateGraph + 节点 + 边可视化、可回滚、可持久化 checkpoint
AutoGen隐式状态机GroupChat + Speaker Selector对话式协作，更灵活但难调试
CrewAI流程式Process: sequential / hierarchical最简单，但表达能力有限
OpenAI Agents SDK转移式Handoffs + TracingAgent 之间直接 handoff，类似接力

三种调度模式

① 中心化（Supervisor）

一个 Manager 智能体调度 N 个 Specialist。对齐 LangChain RouterChain、CrewAI Manager、AutoGen GroupChatManager。

       ┌──────────────┐
       │  Supervisor  │   ← 只判断意图，不干活
       └──────┬───────┘
              ↓ 路由
       ┌──────┼──────┬──────┐
       ▼      ▼      ▼      ▼
     Spec1  Spec2  Spec3  Spec4

② 去中心化（Peer-to-Peer / Handoff）

Agent 之间直接 handoff，无中心调度。对齐 OpenAI Agents SDK 的 Handoffs、AutoGen 的 free-form conversation。

   AgentA ──handoff──▶ AgentB ──handoff──▶ AgentC
   （用户问 A，A 觉得是 B 的事，转给 B，B 干完再转 C）

③ 分层式（Hierarchical）

多层 Supervisor，每层管一段。对齐企业组织架构。适合超大规模（10+ Agent）。

          Top Supervisor
          /     |     \
       Sup1   Sup2   Sup3
       /\     /\     /\
      A B    C D    E F

状态机到底解决什么问题

  • 可视化：流程图可以画出来，团队对齐有共同语言
  • 可调试：每个节点的输入输出有 schema，能单步调试
  • 可回滚：checkpoint 机制，失败可重放到某节点（LangGraph 的核心卖点）
  • 条件分支 / 循环：天然支持，不用 if/else 在 prompt 里硬写
  • 持久化：长任务（多小时）可以中断、恢复、迁移

何时该用状态机

场景推荐理由
固定流程（审核、编排、SOP）显式状态图流程稳定，状态机能可视化、可审计
开放式对话（客服、研究）隐式 / 自由对话流程不可预测，状态机会变成约束
长任务（需要中断恢复）必须用状态机checkpoint 是刚需
简单工具调用不需要直接 ReAct 即可，加状态机过度工程
◆LangGraph 的状态设计 LangGraph 的核心抽象是 StateGraph：节点是函数 (State) → State，边是条件转移。State 是 TypedDict，标量字段覆盖更新，Annotated[list, add] 字段累加追加。这个设计让"状态流转"变得显式、可观测、可持久化。
✓业界共识 从单 Agent 开始。2026 年 multi-agent 工程化门槛已经很低，但良好的架构设计和清晰的 Agent 职责划分，依然是决定系统成败的核心因素。不要为了多智能体而多智能体。识别瓶颈后再拆分。

Hook 机制
Event-driven

问题：Hook 机制是什么？怎么用？业界有哪些做法？

Hook 是事件驱动的回调：在 Agent 生命周期的关键节点（工具调用前 / 后、用户提交、停止等）触发外部脚本或函数。它的核心价值是把横切关注点（审计、安全、自动化、可观测）从主流程剥离。

Claude Code 的六类 Hook（2026）

Hook 类型触发时机典型用途能否拦截
PreToolUse工具执行前拦截危险命令、参数改写、权限校验能（返回非零退出码阻止）
PostToolUse工具执行后自动 lint / 测试、审计日志、结果改写不能拦截，可改结果
UserPromptSubmit用户提交消息前提示词注入检测、敏感词过滤、上下文注入能
StopAgent 想停止时检查任务是否真的完成，强制继续能（阻止停止）
Notification需要通知用户时桌面通知、邮件、IM 推送—
SubagentStop子 Agent 停止时子任务结果校验、聚合能

配置示例

# Claude Code 配置：git commit 前自动跑 lint + test
hooks:
  pretool_use:
    - if: tool.name == "bash" and tool.args matches "git commit.*"
      run: |
        npm run lint
        npm run test
      feedback: true   # 把结果反馈给模型
      blocking: true  # 失败则阻止 commit

  posttool_use:
    - if: tool.name == "edit"
      run: "~/.claude/hooks/auto-format.sh"
      feedback: false  # 后台静默执行

设计原则

  • 非阻塞优先：能 async 就 async，主流程不等 hook。PreToolUse 例外（要拿到结果决定是否继续）
  • 轻量级：单个 hook 应 < 100ms，重的活儿异步推队列
  • 可失败：hook 抛异常不应阻塞主流程（PostToolUse 例外：审计 hook 失败要有告警）
  • 幂等：同一事件可能触发多次（重试），hook 必须幂等，避免重复副作用
  • 显式声明：配置文件而不是代码里硬编码，方便审计和团队协作

Hook 的核心用途

审计与安全

  • PreToolUse 拦截危险命令（rm -rf /）
  • PostToolUse 记录审计日志
  • UserPromptSubmit 检测 prompt injection

自动化

  • PostToolUse 自动 lint / format / 测试
  • Stop 检查任务完整性
  • Notification 推送结果
◆Hook vs Middleware Hook 是事件驱动（被动触发），Middleware 是管道式（主动包裹）。Hook 适合"我不关心主流程，只关心某个事件"的场景；Middleware 适合"我要改请求 / 响应本身"的场景。两者互补，不要混用。

审计与可观测性
Observability

问题：Agent 是黑盒，怎么知道它干了什么？怎么审计？怎么观测？

!为什么传统 APM 不够 传统 APM（New Relic / Datadog）追踪的是函数调用 / HTTP 请求，对 Agent 的"黑盒推理"无能为力——你不知道 Agent 为什么调了某个工具、为什么选了某个参数。Agent 可观测必须把推理过程本身作为一等公民。

三支柱（Three Pillars，从 APM 延伸）

支柱是什么Agent 场景的具体内容
Traces链路追踪每次 LLM 调用、tool call、subagent 调用的链路，含 prompt / response / token / latency
Metrics聚合指标token 消耗、调用次数、成功率、P95 延迟、成本
Logs结构化日志结构化 JSON 日志，含 trace_id / user_id / session_id

Traces 的层级结构

trace（一次用户请求）
├── span: router_agent（意图识别）
│   └── span: llm_call（DeepSeek, 200 tokens, 0.8s）
├── span: strategy_config_agent（参数生成）
│   └── span: llm_call（DeepSeek, 1500 tokens, 2.1s）
├── span: verify_action（Pydantic 校验）
├── span: execute_action（工具执行）
│   ├── span: tool: add_strategy
│   └── span: tool: persist_runtime
└── span: response_format

关键设计要点

  • Trace ID 贯穿：从用户请求到响应，所有 span 共享一个 trace_id，能串联日志、指标、trace
  • 采样策略：正常 1% 全量记录（含 prompt/response），异常 / 失败 100% 记录。Anthropic 内部就是这么做的
  • PII 脱敏：prompt 里可能有用户隐私（身份证 / 手机），落库前必须脱敏
  • 成本归因：每个 Agent / 每次调用烧了多少 token / 多少钱，要能下钻
  • 多模态：图片 / 音频 trace 要能存原文件或 URL，不能只存文本
  • 实时告警：失败率突增、token 消耗异常、延迟 P99 飙升 → 自动告警

各组件选型

Chunking（分块）

  • 固定大小：512-1024 token，简单但可能切断语义
  • 语义分块：按句子 / 段落，保留语义完整
  • 父子分块：小块召回，大块返回（避免上下文不足）
  • 层级分块：章节 / 段落 / 句子多层，可按需取不同粒度
  • Markdown / Code aware：按标题 / 函数边界切，结构信息保留

方案类型适用

检索增强技术

  • Query 改写：用户问题 → LLM 改写成更利于检索的 query
  • HyDE：Hypothetical Document Embeddings，先让 LLM 编答案再检索（提升召回）
  • Multi-query：LLM 生成 N 个变体 query，并行检索后合并去重
  • Reranking：Cohere Rerank / bge-reranker，对召回结果二次排序，提升精度
  • Hybrid Search：BM25（关键词）+ Vector（语义）融合，取长补短
  • Hybrid + RRF：Reciprocal Rank Fusion 融合两路结果

评估闭环（Ragas 四指标）

指标衡量什么理想值
faithfulness答案是否忠于检索内容（不幻觉）> 0.9
answer_relevancy答案是否切题> 0.85
context_precision检索的内容有多少被用上> 0.7
context_recall应该检索到的是否都检索到了> 0.8

生产要点

  • 增量更新：文档变更只重新 embed 受影响的 chunk，避免全量重建
  • 多租户隔离：不同用户 / 团队的知识库要隔离，权限不能串
  • 失败模式监控：检索为空（召回 0 条）、检索噪声（top-1 不相关）都要监控
  • 缓存：相同 query 缓存 embedding 结果 + 检索结果，省成本
  • 引用追溯：答案要标出引用了哪些 chunk，便于核验（必做）
  • 文档版本化：知识库有版本，回滚方便
✓GraphRAG 适用场景 普通向量 RAG 在"实体关系查询"上很弱（如"A 公司的 CEO 之前在哪家公司任职"）。GraphRAG 把实体和关系建成图，能做多跳推理。2026 年 Microsoft GraphRAG 是代表，但构建成本高（需要 LLM 抽实体），适合实体关系密集的领域（金融、医疗、法律）。
◆Vectorless 趋势 2026 年出现的新方向：Vectorless RAG。完全抛弃向量数据库，让 LLM 直接对文档结构（目录 / 标题 / 摘要）推理决定看哪段。适合文档量小（<100 篇）但精度要求极高的场景。代价是 token 消耗大。

总结
问题核心答案
1. Skill 加载三档：静态注册（<10）/ 动态发现（10-50）/ RAG 检索（>50）。协议化 + 权限模型 + 白名单执行
2. Claude 官方七件套：Skills / MCP / Commands / Hooks / Subagents / CLAUDE.md / Plugins。Markdown 即 skill，渐进式披露是核心哲学
3. 80-100 规模三层路由：L1 意图分类 + L2 RAG 工具检索（top-K=5~10）+ L3 Pydantic + 白名单。临界点 10 / 50
4. 多智能体调度LangGraph 显式状态图 / AutoGen 隐式 / CrewAI 流程式 / OpenAI Handoff。状态机适合固定流程 + 长任务，自由对话用隐式
5. 质量评估五位一体：目标-指标-场景-工具-迭代。三层：在线 / 离线 golden set / 单元。Ragas / Langfuse / Braintrust
6. Hook 机制六类：PreToolUse / PostToolUse / UserPromptSubmit / Stop / Notification / SubagentStop。非阻塞 + 轻量 + 幂等
7. 审计可观测三支柱：Traces / Metrics / Logs。Langfuse 最流行，OpenTelemetry 是 2026 趋势。审计 ≠ 可观测：前者事后合规，后者实时运维
8. 知识库RAG 五代演进：Naive → Advanced → Modular → GraphRAG → Vectorless。Ragas 四指标评估闭环
✓整体判断 2026 年 AI Agent 已从"概念很热"走到"工程化阶段"。可观测性成为独立的工程学科，评估从辅助变成主流程，多智能体从噱头变成实用工具。
核心趋势：协议化（MCP / OTel）、渐进式披露（Claude Skills）、状态机回归（LangGraph）、评估闭环（Ragas / Langfuse）。
陷阱：不要为了新技术而新技术。从单 Agent + 静态注册 + golden set 起步，遇到瓶颈再升级。90% 的场景不需要 100 skill + RAG 检索 + 多 Agent 协作。
