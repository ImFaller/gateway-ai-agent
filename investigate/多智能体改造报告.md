# 多智能体改造报告

> 日期：2026-07-03
> 方向：导师指示 —— "1 主智能体调度（语义识别和任务路由）→ 策略配置/策略查询"
> 业界参考：LangChain RouterChain / AutoGen GroupChatManager / CrewAI Manager

## 一、改造背景

导师提出架构方向：**1 主智能体调度（语义识别和任务路由）→ 策略配置/策略查询**。

核心诉求是让对话入口从"单层 LLM 干所有事"演进为"主智能体只识别意图，子智能体专注做具体业务"的多智能体架构。

## 二、改造前现状

对话入口 [chat.py](file:///d:/all%20files/6.29sx/gateway-ai-agent/frontend/routes/chat.py) 的 `_call_langchain_agent` 采用**单层扁平结构**：

- 一次 LLM 调用同时完成两件事：
  1. 识别用户意图（添加策略？查询状态？执行编排？）
  2. 生成具体参数（策略 ID、触发条件、执行步骤等）
- system_prompt 越来越臃肿（40+ 行，包含 4 种操作的完整参数 schema）
- LLM "一心二用"：既要判断意图又要按 schema 生成参数，容易出错
- 加新功能需改动主 prompt，影响面大

执行层 [graph.py](file:///d:/all%20files/6.29sx/gateway-ai-agent/engine/graph.py) 已经是 LangGraph 多智能体编排（match → security/policy/monitor → aggregate），但**对话入口缺少主从结构**。

## 三、业界技术方案

查询业界主流多智能体框架，"1 主调度 + N 子执行"是标准模式：

| 框架 | 主智能体角色 | 对应本项目 |
|---|---|---|
| LangChain RouterChain | Router 链先判断意图，再路由到专用子链 | RouterAgent |
| AutoGen GroupChatManager | Manager 识别语义并分发给专家 Agent | 主调度智能体 |
| CrewAI Manager | Manager 不干活只调度，Specialist 执行 | 子智能体 |

**关键原则**：
- 主智能体 prompt 短、专注（只问意图，不问参数）
- 子智能体各司其职（一个子智能体只做一类业务）
- 简单意图（如查询）可不调 LLM，直接返回动作，节省 token

**不需要配置多个大模型**：主从架构是 prompt 拆分，不是模型拆分。一个 DeepSeek 模型扮演不同角色（不同 prompt）即可。

## 四、改造后架构

```
用户输入
   ↓
┌─────────────────────────────────┐
│  RouterAgent（主智能体）          │  第 1 次 LLM 调用
│  - 语义识别                      │  只返回 {intent, reply}
│  - 任务路由                      │  不生成任何参数
└──────────┬──────────────────────┘
           ↓
┌──────────┴───────────────────────────────┐
│           路由分发（无 LLM）                │
│  intent → 对应子智能体                    │
└──┬──────────────┬───────────────┬────────┘
   ↓              ↓               ↓
┌──────────┐  ┌──────────┐   ┌──────────┐
│Strategy  │  │Strategy  │   │Strategy  │
│Config    │  │Query     │   │Execute   │
│Agent     │  │Agent     │   │Agent     │
│(第2次LLM)│  │(不调LLM) │   │(第2次LLM)│
└──────────┘  └──────────┘   └──────────┘
生成策略参数    直接返回动作    生成执行上下文
(add/delete)   (get_status)   (execute)
```

## 五、改造内容

### 1. 新建 [agents/router_agent.py](file:///d:/all%20files/6.29sx/gateway-ai-agent/agents/router_agent.py)

包含 4 个智能体类：

| 类 | 职责 | LLM 调用 |
|---|---|---|
| RouterAgent（主智能体） | 识别意图，返回 intent | 第 1 次 |
| StrategyConfigAgent（策略配置） | 生成 add/delete 策略参数 | 第 2 次 |
| StrategyQueryAgent（策略查询） | 查询系统状态 | 不调（直接返回动作） |
| StrategyExecuteAgent（策略执行） | 生成执行上下文 | 第 2 次 |

**设计要点**：
- 主智能体 prompt 仅 15 行，只问意图不问参数
- 子智能体 prompt 各自独立，专注一类业务
- 查询类意图跳过 LLM 调用，节省 token
- 所有智能体继承 BaseAgent，统一异常重试 + 降级机制

### 2. 重构 [chat.py](file:///d:/all%20files/6.29sx/gateway-ai-agent/frontend/routes/chat.py) 的 `_call_langchain_agent`

从"单层 LLM"改为"主从两阶段"：

- **第 1 阶段**：调用 RouterAgent 识别意图，得到 `{intent, reply}`
- **第 2 阶段**：根据 intent 路由到对应子智能体
  - `strategy_config` → StrategyConfigAgent（生成策略参数）
  - `strategy_query` → StrategyQueryAgent（直接返回动作）
  - `strategy_execute` → StrategyExecuteAgent（生成执行上下文）
  - `chat` → 直接用主智能体的回复

- 原有的 `_parse_actions` / `_extract_json_blocks` / `_simulate_chat` 保持不变，子智能体复用这些工具函数
- 密码二次确认、Pydantic 参数校验、LangGraph 编排等下游流程完全不变

## 六、前后对比

| 维度 | 改造前 | 改造后 |
|---|---|---|
| LLM 调用次数 | 1 次（意图+参数一起生成） | 1~2 次（主识别意图，子生成参数） |
| 主 prompt 长度 | 40+ 行（4 种操作 schema 全塞） | 15 行（只问意图） |
| 加新功能 | 改主 prompt，影响所有现有功能 | 加一个子智能体类，不动主智能体 |
| 查询类操作 | 仍调 LLM 生成参数 | 跳过 LLM，直接返回动作 |
| 可调试性 | 出错难定位是意图还是参数错 | 两层日志，能定位到哪层出错 |
| 模型配置 | 1 个模型 | 1 个模型（无需多配） |

## 七、可扩展性

主从架构为后续扩展留出空间：

| 未来需求 | 扩展方式 |
|---|---|
| 策略调优 | 新增 StrategyTuneAgent，主智能体 intent 加 strategy_tune |
| 日志分析 | 新增 LogAnalysisAgent |
| 批量操作 | 新增 BatchProcessAgent |
| 不同子智能体用不同模型 | 子智能体独立配 model_config |

## 八、验证方式

改造后测试流程（需重启后端 + 浏览器硬刷新）：

1. **测试意图识别**：输入"你好" → 主智能体应识别为 `chat` 意图，直接回复
2. **测试策略配置**：输入"添加SSH防护策略" → 主智能体识别 `strategy_config` → 子智能体生成策略参数 → 弹密码框 → 执行
3. **测试策略查询**：输入"查看系统状态" → 主智能体识别 `strategy_query` → 直接返回 `get_system_status` 动作（不调第二次 LLM）
4. **测试策略执行**：输入"执行一次流量审查" → 主智能体识别 `strategy_execute` → 子智能体生成 context → 执行编排

后端控制台会打印分层日志：
```
[主智能体] 意图: strategy_config, 回复: 好的，正在为您配置策略...
[策略配置子智能体] 生成动作: [{'action': 'add_strategy', 'params': {...}}]
```

## 九、总结

本次改造将对话入口从"扁平单层 LLM"升级为"1 主调度 + N 子执行"的多智能体架构，符合导师提出的"主智能体做语义识别和任务路由，再分发到策略配置/策略查询"的方向。

改造借鉴业界 LangChain RouterChain / AutoGen GroupChatManager 模式，主智能体专注意图识别、子智能体各司其职，提升了架构清晰度、可维护性和可扩展性，且无需配置多个大模型。
