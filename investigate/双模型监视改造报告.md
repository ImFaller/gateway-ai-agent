# 双模型监视改造报告（LLM-as-a-Judge）

## 一、改造背景

前期已完成单模型主从架构（RouterAgent 主调度 + 子智能体执行）和 Hermes 格式校验（Verifier）。但 LLM 输出仍存在两类风险：

1. **语义错误**：LLM 把"添加"理解成"删除"，或参数与用户意图不符
2. **危险操作**：用户说"删除所有策略"，LLM 真的生成批量删除动作

格式校验（Hermes Verifier）只能看字段是否齐全，无法判断"动作是否合理"。业界标准做法是引入第二个模型做语义审查。

## 二、业界技术方案

### LLM-as-a-Judge（OpenAI Evals 官方推荐）

OpenAI 在 Evals 指南中明确推荐：用 GPT-4 评判 GPT-3.5 的输出。核心思想是 **"让一个模型评判另一个模型的输出"**。

### Hermes MoA（Mixture of Agents）

Hermes Agent 的 MoA 模式：多个模型独立给出判断，再聚合结果。本项目采用其简化版——主模型生成动作，监视模型审查动作。

### LangChain CriteriaEvalChain

LangChain 提供基于准则的 LLM 评估链，可指定审查维度（如"是否危险"、"是否符合意图"）。本项目的 VerifierAgent 借鉴这一思路。

## 三、改造内容

### 1. 后端：ModelConfig 加 role 字段

`frontend/routes/models.py` 的 `ModelConfig` 新增字段：

```python
role: str = Field(default="primary", description="模型角色：primary/verifier/both")
```

- `primary`：主模型（生成动作）
- `verifier`：监视模型（审查动作）
- `both`：两用（既可作主也可作监视）

### 2. 后端：新增 settings.py 路由

`frontend/routes/settings.py` 提供 `/api/v1/settings` 接口：

- `verify_strength`：监视强度（off/warn_only/high_risk_only/strong）
- `verifier_model_id`：监视模型ID

持久化到 `gateway_ai_settings.json`。

### 3. 后端：新增 VerifierAgent

`agents/router_agent.py` 新增 `VerifierAgent` 类：

- 继承 BaseAgent，获得 CrewAI 三要素（role/goal/backstory）
- `role = "动作审查员"`
- `verify(user_message, action, verifier_model_config)` 方法
- 用低温（temperature=0.1）保证输出稳定
- 调用失败降级为通过（避免监视模型故障阻塞主流程）

审查维度：
1. 动作类型是否匹配用户意图
2. 参数是否合理
3. 是否有越权或危险操作
4. 用户原始请求与动作是否一致

### 4. 后端：chat.py 主流程插入第 4 层监视

`_call_langchain_agent` 流程从 3 层扩展为 4 层：

```
第 1 层：RouterAgent 识别意图
第 2 层：子智能体生成动作
第 3 层：Hermes Verifier 格式校验（已有）
第 4 层：LLM-as-a-Judge 语义审查（新增）  ← 本次改造
```

`_run_llm_judge` 函数按强度策略执行：

| 强度 | 行为 |
|---|---|
| `off` | 不调监视模型 |
| `warn_only` | 监视不通过时只警告，仍执行 |
| `high_risk_only` | 仅 add/delete_strategy 调监视，拦截不通过的动作 |
| `strong` | 所有动作都调监视，全部需一致同意 |

### 5. 前端：设置面板加监视配置

`admin.html` 设置页新增「双模型监视」卡片：

- **监视强度下拉**：4 个选项（关闭/仅警告/仅高危审查/强一致）
- **监视模型下拉**：自动从模型列表筛选 role=verifier/both 的模型
- 改动即时保存到后端

### 6. 前端：模型管理加角色标识

- 模型列表新增「角色」列，显示 `主`/`监视`/`主+监` 标签
- 添加/编辑模型表单新增「模型角色」下拉
- `doSaveModel` 发送 role 字段

## 四、用户使用流程

### 配置流程

1. 在「模型管理」添加第二个模型（如 GPT-4o-mini）
   - 表单底部「模型角色」选「监视模型」
2. 在「设置」→「双模型监视」
   - 监视强度选「仅高危审查」（推荐）
   - 监视模型选刚添加的模型
3. 保存设置

### 使用效果

**正常情况**（主监视都同意）：
```
用户：添加 SSH 防护策略
[主模型] 生成 add_strategy 动作
[监视模型] 审查通过
✅ 策略已添加
```

**异常情况**（监视模型拦截）：
```
用户：删除所有策略
[主模型] 生成 delete_strategy 动作
[监视模型] ❌ 拒绝：删除范围过大，建议人工确认
❌ 监视模型拒绝执行（delete_strategy）：删除范围过大
```

**仅警告模式**：
```
用户：添加危险策略
[主模型] 生成动作
[监视模型] ⚠️ 警告：参数异常
⚠️ 监视模型警告：参数异常
（动作仍执行，但用户收到警告）
```

## 五、成本估算

假设每天 100 次操作，其中 20 次高危（high_risk_only 模式）：

- 监视模型 token 消耗：20 × ~500 tokens = 10K tokens/天
- 用 gpt-4o-mini（$0.15/1M tokens）：约 $0.0015/天
- 几乎零成本

## 六、与现有改造的关系

| 改造 | 层级 | 作用 |
|---|---|---|
| 主从架构（RouterAgent） | 对话层 | 意图识别 + 任务路由 |
| CrewAI Role-based | 智能体定义 | 角色化 prompt |
| Hermes Verifier | 格式校验 | 字段合法性 |
| **LLM-as-a-Judge**（本次） | **语义审查** | **动作合理性** |

四者互补：主从架构管"谁干"，Role-based 管"怎么干"，Hermes 管"格式对不对"，LLM-as-a-Judge 管"该不该干"。

## 七、验证方式

1. 重启后端：`python -m uvicorn system.app:app --host 0.0.0.0 --port 8099`
2. 浏览器硬刷新：Ctrl+F5
3. 在「模型管理」添加第二个模型，角色选「监视模型」
4. 在「设置」配置监视强度和监视模型
5. 测试：
   - 正常添加策略 → 监视模型通过 → 执行成功
   - 输入"删除所有策略" → 监视模型拦截 → 显示拒绝原因
6. 后端日志应看到：
   ```
   [主智能体] 意图: strategy_config
   [策略配置子智能体] 生成动作: [...]
   [监视模型] 通过: add_strategy - 动作合理
   ```

## 八、总结

本次改造借鉴业界 **LLM-as-a-Judge** 模式（OpenAI Evals / Hermes MoA / LangChain CriteriaEvalChain），实现双模型交叉验证机制：

- 用户可选 4 种监视强度（关闭/仅警告/仅高危/强一致）
- 用户可选任意模型作监视模型（role=verifier/both）
- 监视模型独立审查主模型输出，拦截危险或错误操作
- 监视失败时降级为通过，避免单点故障
- 成本几乎为零（仅高危模式每天约 $0.0015）

改造遵循"小步快跑"原则，未引入新依赖，与现有主从架构、CrewAI 角色化、Hermes 格式校验完全兼容。
