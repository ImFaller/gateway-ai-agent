⚠️ 二、改造前现状
单层扁平 LLM 架构的问题

对话入口的单层结构

对话入口 _call_langchain_agent 采用单层扁平结构：一次 LLM 调用同时完成两件事——识别用户意图 + 生成具体参数。

用户消息
↓
单次 LLM 调用（意图 + 参数）
prompt 40+ 行
↓
返回 action JSON
🔴 改造前的 4 个问题
  • system_prompt 越来越臃肿：40+ 行，包含 4 种操作的完整参数 schema
  • LLM "一心二用"：既要判断意图又要按 schema 生成参数，容易出错
  • 加新功能影响面大：需改动主 prompt，可能影响现有识别
  • 查询类也调 LLM：本可固定返回的动作仍消耗 token

执行层已经领先一步

执行层 graph.py 已经是 LangGraph 多智能体编排（match → security/policy/monitor → aggregate），但对话入口缺少主从结构，导致整条链路头重脚轻。

🌐 三、业界技术方案
主流多智能体框架的"1 主 + N 子"标准模式

业界主流框架对照

框架主智能体角色对应本项目
LangChain RouterChainRouter 链先判断意图，再路由到专用子链RouterAgent
AutoGen GroupChatManagerManager 识别语义并分发给专家 Agent主调度智能体
CrewAI ManagerManager 不干活只调度，Specialist 执行子智能体
✅ 关键原则
  • 主智能体 prompt 短、专注（只问意图，不问参数）
  • 子智能体各司其职（一个子智能体只做一类业务）
  • 简单意图（如查询）可不调 LLM，直接返回动作，节省 token

📝 五、改造内容
新增 router_agent.py + 重构 chat.py

1. 新建 agents/router_agent.py

包含 4 个智能体类：

类职责LLM 调用
RouterAgent（主）识别意图，返回 intent第 1 次
StrategyConfigAgent生成 add/delete 策略参数第 2 次
StrategyQueryAgent查询系统状态不调（直接返回动作）
StrategyExecuteAgent生成执行上下文第 2 次
💡 设计要点
  • 主智能体 prompt 仅 15 行，只问意图不问参数
  • 子智能体 prompt 各自独立，专注一类业务
  • 查询类意图跳过 LLM 调用，节省 token
  • 所有智能体继承 BaseAgent，统一异常重试 + 降级机制

2. 重构 chat.py 的 _call_langchain_agent

从"单层 LLM"改为"主从两阶段"：

  • 第 1 阶段：调用 RouterAgent 识别意图，得到 {intent, reply}
  • 第 2 阶段：根据 intent 路由到对应子智能体

🎯 九、总结
符合导师方向，借鉴业界成熟模式

本次改造将对话入口从"扁平单层 LLM"升级为"1 主调度 + N 子执行"的多智能体架构，符合导师提出的"主智能体做语义识别和任务路由，再分发到策略配置/策略查询"的方向。

改造借鉴业界 LangChain RouterChain / AutoGen GroupChatManager 模式，主智能体专注意图识别、子智能体各司其职，提升了架构清晰度、可维护性和可扩展性，且无需配置多个大模型。
