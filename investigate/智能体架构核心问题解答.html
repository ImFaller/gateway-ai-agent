Context 过长分散注意力
已解决

问题：单个 prompt 塞太多内容（系统说明 + 工具描述 + 历史对话 + 用户输入），模型注意力被稀释，关键信息容易被忽略。

解决方法：主从智能体架构

对齐业界 LangChain RouterChain / AutoGen GroupChatManager / CrewAI Manager 共识。一个轻量主智能体只做意图识别，重的参数生成交给专门子智能体。

多 Skill 如何精准调用
已解决

问题：当 skill 多了，LLM 可能选错（用户说"加策略"却调用了 delete_strategy）。

解决方法：三层防线

第 1 层：意图分类

主智能体先把用户消息归到 5 类意图，每个意图只对应 1-2 个 skill，缩小选择空间。

INTENT_CONFIG  = "strategy_config"    # → 只可能调 add_strategy / delete_strategy
INTENT_QUERY   = "strategy_query"     # → 只可能调 get_system_status
INTENT_EXECUTE = "strategy_execute"   # → 只可能调 execute_strategy
INTENT_RESTORE = "strategy_restore"  # → 只可能调 restore_strategy（恢复已删策略）
INTENT_CHAT     = "chat"              # → 不调任何 skill

第 2 层：SKILL_REGISTRY 白名单

LLM 返回 action 名后由代码查找，不是 LLM 直接执行。未注册的动作直接拒绝。

SKILL_REGISTRY = {
    "get_system_status": {"permission": "allow", "args_schema": None},
    "add_strategy":      {"permission": "ask",   "args_schema": AddStrategyArgs},
    "delete_strategy":   {"permission": "ask",   "args_schema": DeleteStrategyArgs},
    "execute_strategy":  {"permission": "ask",   "args_schema": ExecuteStrategyArgs},
    "restore_strategy":  {"permission": "ask",   "args_schema": RestoreStrategyArgs},
}

skill = SKILL_REGISTRY.get(action)
if not skill:
    logger.warning(f"[审计] 策略:拒绝未注册 action={action} result=fail")
    return {"status": "error", "message": f"未注册的操作: {action}"}
!三层白名单同步原则 新增 skill 必须同步加到 三处，缺一会导致 Verifier 误判格式错误：
  • SKILL_REGISTRY — 总注册表（permission + args_schema）
  • PROTECTED_ACTIONS — 需密码二次确认的高危动作集合
  • ALLOWED_ACTIONS（Hermes Verifier）— 动作格式校验白名单

第 3 层：嵌套 Pydantic 校验

LLM 即使选对了 skill，参数也得过 schema 校验。例如 value 必须是 Union[str, list[str]]，port 必须 0-65535，operator 必须在白名单内。

业界框架选 skill 方式项目对比
LangChain ReActLLM 在所有 tools 中选skill 多了易选错
CrewAI Manager1 Manager 调度 N Specialist项目做法一致
项目意图分类先收窄 + 白名单 + Pydantic错误率低

策略操作的日志
已解决

问题：策略增删改执行必须留痕，便于事后审计、追责、回滚。

解决方法：统一 [审计] 前缀 + 五路由全覆盖 + 前端可视化

文件覆盖的操作
frontend/routes/auth.py登录、改密、二次确认（成功/失败）
frontend/routes/models.py模型增/改/删/启用切换（含重名/连接失败）
frontend/routes/settings.py设置更新（监视强度、主/监视模型）
frontend/routes/api.py策略增/删/执行 + 回收站恢复 + 永久删除（REST API，含失败路径）
frontend/routes/chat.py智能体动作执行：add/delete/execute/restore + 批量删除拒绝 + 重复添加（duplicate/duplicate_content）+ 密码错误

新增审计场景（对话流）

# 批量删除意图被拦截
[审计] 策略:删除 target=__all__ result=fail reason=拒绝批量删除

# 策略恢复成功
[审计] 策略:恢复 target=ssh_protection result=ok reason=从回收站还原

# 内容重复被拦
[审计] 策略:添加 target=custom result=duplicate_content reason=与 ssh_protection 内容一致

# 密码二次确认失败
[审计] 认证:二次确认 action=delete_strategy result=fail reason=密码错误

统一格式

[审计] 类别:动作 target=xxx result=ok/fail reason=xxx

设计原则

  • 失败也要记：参照 Linux audit.log / SSH auth.log 共识，密码错误、删除找不到、参数错误等失败路径全部记录
  • 读操作不记审计：参照 Kubernetes audit log / AWS CloudTrail 共识，list/get 类查询不记，避免日志爆炸
  • 敏感信息不进日志：API Key、密码明文都不记，符合 OWASP 日志安全要求
  • 惰性获取 logger：函数内导入 from system.app import app_state，避免循环依赖
✓前端可视化 侧边栏"审计日志"页面支持：类别过滤（策略/模型/设置/认证/安全）、级别过滤、关键字搜索、5 秒自动刷新、result=fail 红色高亮。表格风格对齐 Linux audit.log / kubectl describe。

入口适用场景命令/操作

恢复路径

出错场景恢复方法
策略误删（30 天内）回收站恢复：策略管理页面 → 回收站弹窗 → 选中 → 恢复；或调 POST /api/v1/strategies/{id}/restore
策略误删（对话恢复）对智能体说"恢复 SSH 防护策略"/"恢复刚才删的策略"，RestoreStrategyAgent 路由到 restore_strategy skill，支持 __all__ 批量恢复
策略误删（超 30 天）重启从 strategies.yaml 重新加载基础策略；运行时策略从 gateway_ai_runtime_strategies.yaml 持久化恢复
策略添加失败审计日志记了 result=fail reason=xxx，按 reason 修正参数后重试
策略重复添加kubectl apply 语义：内容完全一致返回 duplicate 不写；同 ID 内容不同覆盖为 updated；不同 ID 内容相同返回 duplicate_content 拒绝
批量删除误操作三层防御拦截：①LLM prompt 不返回 action ②Verifier 拒绝通配符 ③execute_action 拒绝 __all__/all/*，根本不会删
模型配错编辑 gateway_ai_models.json 改回正确值
设置改坏编辑 gateway_ai_settings.json
密码忘记跑 python reset_password.py 重置为默认 123456（对齐 MySQL --skip-grant-tables），重置后强制改密
密码输错（≤3 次）前端 3 次重试，错误不缓存，正确才进 sessionStorage；3 次失败锁定该次操作
execute_strategy 失败抛错并记 ERROR 审计，不影响其他策略
✓回收站机制（软删除） 对齐 Slack / Google Drive / Notion 共识。remove_strategy 不真删，而是把策略移入 _trash 列表并持久化到 gateway_ai_runtime_trash.yaml。保留 30 天，到期由 purge_strategy 永久删除。
不对称设计：删除禁止批量（危险方向），恢复允许批量 __all__（安全方向）。
✓持久化已实现 运行时策略持久化到 gateway_ai_runtime_strategies.yaml（与 strategies.yaml 分离，对齐 K8s 默认 ConfigMap + 用户自定义资源模式）。add/remove/update 后同步写盘，写失败不阻塞主流程。

Skill 如何加载
已解决

问题：skill 是怎么从代码变成可被智能体调用的？加载时机和机制是什么？

解决方法：字典静态注册

适合 skill 数量少的场景。模块导入时字典就在内存里，O(1) 查找，没有反射开销。

# 技能注册表：name → {permission, args_schema}
# permission: allow=自动执行, ask=需确认, deny=禁止（CrewAI 模式）
SKILL_REGISTRY = {
    "get_system_status": {
        "permission": "allow",          # CrewAI 权限模型
        "args_schema": None,            # LangChain args_schema 模式
    },
    "add_strategy": {
        "permission": "ask",
        "args_schema": AddStrategyArgs,  # Pydantic 模型
    },
    "delete_strategy": {
        "permission": "ask",
        "args_schema": DeleteStrategyArgs,
    },
    "execute_strategy": {
        "permission": "ask",
        "args_schema": ExecuteStrategyArgs,
    },
    "restore_strategy": {
        "permission": "ask",
        "args_schema": RestoreStrategyArgs,  # 恢复已删策略
    },
}

Skill 数量多少有什么影响
需关注

问题：skill 多了对系统有什么影响？什么时候需要换架构？

!当前状态 当前 5 个 skill（含 restore_strategy），远未到瓶颈。但需要知道临界点，提前规划。

对各环节的影响

影响维度严重程度说明
主智能体 prompt 长度低主智能体只列意图类别（5 类），不列 skill 详情，skill 数量不影响主 prompt
子智能体 prompt 长度中每个子智能体要在 prompt 里说明自己负责的 skill，skill 多了 prompt 变长
路由准确性中意图分类变难（如"删除策略"和"删除模型"可能都归到 strategy_config）
维护成本高每加一个 skill 要同步更新三处白名单（SKILL_REGISTRY + PROTECTED_ACTIONS + ALLOWED_ACTIONS）+ schema + 子智能体 prompt + 测试
LLM 选错概率中skill 越多，LLM 选错的概率越高（虽然项目用意图分类已大幅降低）

业界共识的临界点

Skill 数量推荐做法业界参考
< 10 个静态注册（项目当前做法）LangChain 默认
10-50 个工具描述压缩 + 分组LangChain ToolRetriever
> 50 个RAG 检索工具描述（只把相关的 top-K 喂给 LLM）LangChain VectorStoreRetrieverTool

未来扩展建议（超过 10 个时）

  1. 把意图分类从 5 类扩到更细（如 model_config / auth_config 等）
  2. 给每个子智能体单独的 prompt 文件，避免一个文件管所有
  3. 考虑用 LangChain ToolRetriever 模式（按用户消息检索 top-3 相关 skill）
  4. 把三处白名单的同步做成单点注册函数，避免漏改（当前靠人工约定）
